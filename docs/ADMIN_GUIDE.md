# GasMeter Pro — инструкция администратора

Документ описывает порядок установки, проверки, резервного копирования и сопровождения пилотной версии GasMeter Pro.

## 1. Назначение администратора

Администратор отвечает за:

- запуск приложения;
- проверку состояния backend/frontend;
- контроль целостности рабочей папки данных;
- резервное копирование базы и скан-копий МИ;
- обновление версии приложения;
- контроль прохождения тестов;
- передачу инцидентов разработчику.

## 2. Структура проекта

```text
GasMeter Pro
├─ backend/
│  ├─ app/
│  ├─ tests/
│  ├─ requirements.txt
│  └─ data/
│     ├─ gasmeter.db
│     ├─ method_documents/
│     └─ reports/
├─ frontend/
│  ├─ src/
│  └─ package.json
├─ docs/
├─ scripts/
│  ├─ validate.ps1
│  └─ validate.sh
└─ compose.yaml
```

## 3. Рабочие данные

Рабочие данные находятся в:

```text
backend/data/
```

Внутри хранятся:

| Путь | Назначение |
|---|---|
| `backend/data/gasmeter.db` | база SQLite |
| `backend/data/method_documents/` | PDF-скан-копии МИ |
| `backend/data/reports/` | сформированные протоколы PDF/DOCX |

Эта папка не должна удаляться при обновлении приложения.

## 4. Запуск через Docker

Основной способ запуска пилотной версии:

```bash
docker compose -f compose.yaml up --build
```

После запуска открыть:

```text
http://localhost:5173
```

Backend API:

```text
http://localhost:8000
```

## 5. Проверка состояния системы

Проверка backend:

```text
http://localhost:8000/health
```

Ожидаемый ответ:

```json
{
  "status": "ok",
  "application": "GasMeter Pro"
}
```

Диагностика backend и базы:

```text
http://localhost:8000/api/system/info
```

Проверяются:

- версия приложения;
- версия схемы БД;
- ожидаемая версия схемы;
- путь к базе;
- наличие файла базы.

## 6. Полная локальная проверка

Windows PowerShell:

```powershell
.\scripts\validate.ps1
```

Linux / macOS:

```bash
bash scripts/validate.sh
```

Скрипт выполняет:

1. Backend tests.
2. Frontend build.
3. Docker compose build.

Если хотя бы один этап падает, версию нельзя передавать в пилотную эксплуатацию.

## 7. Резервное копирование

Перед обновлением приложения необходимо сохранить папку:

```text
backend/data/
```

Рекомендуемая структура архива:

```text
backup_YYYY-MM-DD_HH-MM/
├─ gasmeter.db
├─ method_documents/
└─ reports/
```

Минимальная периодичность резервного копирования на пилоте:

| Режим | Периодичность |
|---|---|
| Активный пилот | ежедневно |
| Редкое использование | перед каждым обновлением и не реже 1 раза в неделю |
| Перед загрузкой новой партии МИ | обязательно |

## 8. Восстановление из резервной копии

1. Остановить приложение.
2. Сохранить текущую папку `backend/data/` как аварийную копию.
3. Заменить содержимое `backend/data/` из резервной копии.
4. Запустить приложение.
5. Проверить `/api/system/info`.
6. Открыть историю и проверить наличие расчётов.
7. Проверить доступность PDF-скан-копий МИ.

## 9. Обновление версии

Перед обновлением:

1. Сохранить `backend/data/`.
2. Выполнить `scripts/validate.ps1` или `scripts/validate.sh` на новой версии.
3. Проверить changelog / список изменений.
4. Запустить приложение.
5. Проверить `/health` и `/api/system/info`.
6. Открыть интерфейс.
7. Проверить одну МИ, один расчёт, один экспорт PDF/DOCX.

## 10. Контроль целостности МИ

Для каждой версии МИ должен быть зафиксирован SHA-256 PDF-файла.

При сомнениях:

1. Открыть журнал истории.
2. Выбрать расчёт.
3. Нажать `Проверить SHA-256 МИ`.
4. Проверить статус:
   - `valid` — файл соответствует;
   - `changed` — файл изменён;
   - `missing` — файл отсутствует.

При статусе `changed` или `missing` расчёт нельзя использовать как воспроизводимый без дополнительной проверки.

## 11. Действия при ошибках

| Симптом | Возможная причина | Действие |
|---|---|---|
| Frontend не открывается | контейнер frontend не запущен | проверить Docker Compose |
| Backend недоступен | контейнер backend не запущен | проверить `/health`, логи контейнера |
| База не найдена | отсутствует `gasmeter.db` | проверить `backend/data/` |
| Не открывается PDF МИ | файл отсутствует в `method_documents` | проверить путь и восстановить из backup |
| SHA-256 изменился | файл был заменён | проверить версию МИ и историю изменений |
| PDF/DOCX не формируется | ошибка backend или зависимостей | запустить backend tests |

## 12. Логи и диагностика

Для Docker Compose:

```bash
docker compose -f compose.yaml logs
```

Для конкретного сервиса:

```bash
docker compose -f compose.yaml logs backend
```

```bash
docker compose -f compose.yaml logs frontend
```

## 13. Минимальный чек-лист передачи версии в пилот

Перед передачей версии пользователю должны быть выполнены пункты:

- [ ] приложение запускается через Docker;
- [ ] `/health` возвращает `ok`;
- [ ] `/api/system/info` показывает корректную БД;
- [ ] backend tests проходят;
- [ ] frontend build проходит;
- [ ] docker build проходит;
- [ ] загружена хотя бы одна реальная МИ;
- [ ] PDF МИ имеет SHA-256;
- [ ] сохранён один расчёт;
- [ ] расчёт загружается из истории;
- [ ] PDF/DOCX формируется из текущей формы;
- [ ] PDF/DOCX формируется из истории;
- [ ] SHA-256 проверяется из истории.

## 14. Ограничения администратора

Администратор не должен:

- вручную редактировать `gasmeter.db` без резервной копии;
- заменять PDF-файлы МИ без создания новой версии МИ;
- удалять `backend/data/`;
- переносить приложение в продакшн без верификации расчётов по реальным МИ;
- отключать тесты перед передачей версии.

## 15. Что требуется перед продакшн

До промышленного применения необходимо дополнительно реализовать:

- роли пользователей;
- журнал действий пользователей;
- механизм утверждения версии МИ;
- блокировку утверждённых версий;
- централизованное резервное копирование;
- регламент обновления МИ;
- акт верификации расчётов;
- утверждённую матрицу МИ → формула → тест.
