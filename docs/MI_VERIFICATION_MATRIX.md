# Матрица верификации МИ → формула → тест

Документ фиксирует порядок доведения GasMeter Pro до пилотной готовности и дальнейшей промышленной верификации.

Цель матрицы — исключить ситуацию, когда расчётный шаблон существует в программе, но не подтверждён конкретными пунктами методики измерений и контрольными примерами.

## Правило допуска МИ в пилот

Методика измерений считается готовой к пилотной эксплуатации только при выполнении всех условий:

1. В библиотеке МИ создана активная версия методики.
2. К версии МИ загружена PDF-скан-копия.
3. Для PDF зафиксирован SHA-256.
4. Выбран расчётный шаблон.
5. Заполнены диапазоны применимости Q/P/T.
6. Указан предел допустимой суммарной погрешности / расширенной неопределённости.
7. По методике подготовлены контрольные примеры.
8. Контрольные примеры проходят автоматическую проверку.
9. Результат ручной проверки совпадает с результатом программы в заданном допуске.
10. Сформирован протокол расчёта PDF/DOCX.

## Статусы готовности

| Статус | Значение | Допуск к пилоту |
|---|---|---|
| draft | Методика заведена, но расчёт не подтверждён | Нет |
| formula_mapped | Формулы сопоставлены с пунктами МИ | Нет |
| test_cases_added | Добавлены контрольные примеры | Ограниченно |
| verified | Контрольные примеры пройдены | Да |
| approved | Методика утверждена владельцем процесса | Да |
| archived | Версия МИ выведена из применения | Нет |

## Обязательный состав контрольного примера

Каждый контрольный пример должен содержать:

| Поле | Описание |
|---|---|
| `name` | Название контрольного примера |
| `input_data.line` | Параметры измерительной линии |
| `input_data.errors` | Составляющие погрешности / неопределённости |
| `input_data.context` | Рабочая точка, давление, температура, Z-факторы |
| `expected_result.delta_total` | Ожидаемая суммарная погрешность / неопределённость |
| `expected_result.status` | Ожидаемый статус: pass / warn / fail |
| `tolerance` | Допуск сравнения результата |
| `source_reference` | Пункт МИ или расчётный лист, откуда взят пример |

## Минимальный набор контрольных примеров на одну МИ

| № | Сценарий | Цель проверки |
|---:|---|---|
| 1 | Номинальная рабочая точка | Подтвердить базовый расчёт |
| 2 | Минимальный расход Qmin | Проверить нижнюю границу применимости |
| 3 | Максимальный расход Qmax | Проверить верхнюю границу применимости |
| 4 | Граничное давление / температура | Проверить PTZ-пересчёт |
| 5 | Превышение допустимого предела | Проверить статус fail |

## Критерии готовности 90%

Программа считается готовой примерно на 90% для пилота, когда выполнено:

| Блок | Критерий | Статус |
|---|---|---|
| Запуск | Docker Compose запускает frontend/backend | частично готово |
| Интерфейс | Экранная форма, история, библиотека МИ, выгрузка | готово |
| История | Загрузка расчёта, PDF/DOCX из истории, SHA-256 | готово |
| Шаблоны | Реестр шаблонов и автотесты | готово |
| МИ | 2–3 реальные МИ полностью заведены | требуется |
| Контрольные примеры | 3–5 примеров на каждую пилотную МИ | требуется |
| Верификация | Ручная сверка с МИ и протоколом | требуется |
| Документация | Инструкция пользователя и администратора | требуется |

## Порядок верификации новой МИ

1. Загрузить PDF методики в библиотеку МИ.
2. Зафиксировать SHA-256.
3. Создать новую версию МИ.
4. Выбрать расчётный шаблон.
5. Заполнить диапазоны применимости.
6. Сопоставить пункты МИ с формулами в коде.
7. Добавить контрольные примеры.
8. Запустить автоматическую проверку контрольных примеров.
9. Сохранить расчёт в историю.
10. Выгрузить PDF/DOCX протокол.
11. Проверить SHA-256 из истории.
12. Присвоить статус `verified` или вернуть на доработку.

## Открытые вопросы перед продакшн

- Нужны роли пользователей: администратор, метролог, инженер, наблюдатель.
- Нужен журнал действий пользователя.
- Нужен регламент резервного копирования `backend/data`.
- Нужен механизм утверждения версии МИ.
- Нужна блокировка изменения утверждённых версий МИ.
- Нужен импорт/экспорт справочника СИ.
- Нужна промышленная сверка формул по каждой реальной МИ.
